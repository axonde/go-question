# ADR-002: Базовая стратегия Dependency Injection

## Status

Accepted

## Date

2026-04-03

## Context

- Проект развивается в формате интенсива, время на архитектурные итерации ограничено.
- При росте числа зависимостей ручное внедрение через конструкторы по всей кодовой базе снижает читаемость и повышает вероятность ошибок.
- Нужна единая точка инициализации зависимостей, чтобы ускорить добавление фич и упростить замену реализаций.
- Прогнозируемый риск: разрастание DI-конфигурации в один трудночитаемый файл.

## Decision

- Использовать пакет GetIt как базовый механизм DI.
- Хранить регистрацию зависимостей в отдельном слое контейнера, а не размазывать по feature-коду.
- Разделять регистрации по модулям (минимум: core, auth, event), чтобы избежать монолитного контейнера.
- Новые зависимости добавлять только через модульные registration-функции; точечные регистрации в произвольных местах запрещены.

## Consequences

### Positive

- Единый и предсказуемый способ подключения зависимостей.
- Проще заменять реализации и тестировать feature-модули.
- Быстрее онбординг: у команды одна точка входа в DI.

### Negative

- При слабой дисциплине контейнер все равно может стать перегруженным.
- Service Locator скрывает часть зависимостей и требует строгих правил использования.

### Revisit triggers

- DI-конфигурация перестает быть модульной и превращается в один длинный файл.
- Добавление новой фичи требует правок в несвязанных DI-секциях.
- В PR регулярно появляются ошибки из-за пропущенной/неверной регистрации зависимостей.
